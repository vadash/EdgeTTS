# Design: Assign Phase QA Pass

## Objective
Replace the 3-way temperature voting consensus in the Speaker Assignment (Assign) phase with a new sequential Quality Assurance (QA) pass. This pass will review a draft set of assignments generated by a standard Assign call, explicitly check for common LLM mistakes (vocative traps, missed action beats), and output a corrected, full set of assignments. 

## Approach
Instead of running 3 parallel requests and calculating a mathematical consensus, the system will run a standard Assign request to generate a "draft". It will then feed that draft, alongside the original text and context, into a new QA prompt. The QA LLM will output the final `Record<string, string>` assignments.

The existing `useVoting` toggle in the settings will be kept internally to minimize state migrations, but its UI labels will be updated to reflect the new "QA Pass" behavior.

## Architecture & Data Flow

### 1. `LLMVoiceService.processAssignBlock`
The execution flow within `processAssignBlock` will be refactored:
1. **Always** execute a standard Assign API call using the configured temperature/settings.
2. If `this.options.useVoting` is `false`, return the parsed results immediately (status quo).
3. If `this.options.useVoting` is `true`:
   - Take the raw JSON response from the first call (the "draft").
   - Construct the QA prompt using the new `buildQAPrompt` function, injecting the draft assignments as a separate XML block.
   - Execute a second API call to the QA prompt using the exact same configured temperature and settings.
   - Parse the QA response using the existing `AssignSchema`.
   - Return the QA-corrected results.

### 2. Prompt Engineering (`src/config/prompts/qa/*`)
A new prompt directory will be created to mirror the existing stages (`extract`, `merge`, `assign`).

* **Role (`qa/role.ts`):** You are an expert dialogue editor and quality assurance bot. Your job is to review a draft speaker attribution for a text, find mistakes, and output the corrected mapping.
* **Rules (`qa/rules.ts`):** Explicitly instruct the model to look for common LLM errors:
  - *Vocative Trap:* Did the draft assign the quote to the person being spoken to?
  - *Action Beats:* Did the draft miss an action beat indicating a different speaker?
  - *Narration:* Did the draft accidentally assign a speaker code to a purely narrational paragraph? (Remove it).
  - *Missing Dialogue:* Did the draft miss a paragraph containing dialogue? (Add it).
* **Builder (`qa/builder.ts`):** 
  - Takes all standard Assign arguments *plus* the draft assignments.
  - Injects `<draft_assignments>` as a separate XML block at the end of the user prompt.
  - Injects overlap sentences exactly as they are passed to the standard Assign prompt.
* **Schema:** We will reuse the existing `AssignSchema` and `ASSIGN_SCHEMA_TEXT` since the output format (the full set of assignments) remains identical.
* **Examples (`qa/examples/en.ts`):** 2-3 few-shot examples showing a flawed draft being corrected in the reasoning field, followed by the perfect output.

## Code Modifications

1. **`src/services/llm/LLMVoiceService.ts`**
   - Remove the `VOTING_TEMPERATURES` constant and loop.
   - Implement the sequential `Assign -> QA` flow.
   - Save debug logs for *both* passes (e.g., `assign_draft` and `assign_qa`) if logging is enabled.

2. **`src/services/llm/votingConsensus.ts`**
   - Remove the `majorityVote` function (it is no longer used).
   - Keep `buildMergeConsensus` (the Merge phase still uses 5-way voting).

3. **`src/config/prompts/qa/*`**
   - Scaffold the new QA prompt directory and export it via `src/config/prompts/index.ts`.

4. **`src/i18n/*.json`**
   - Update `settings.llm.useVoting` (and related descriptions) in `en.json` and `ru.json` from "Use Voting Consensus (3x API Cost)" to something like "Enable QA Pass (2x API Cost)". Internal state keys remain `useVoting`.

## Trade-offs
* **Cost/Speed:** Reduces API calls from 3 to 2 for this phase, improving speed and lowering costs while structurally improving accuracy through targeted critique.
* **Latency:** Because the calls are sequential rather than parallel, the wall-clock time for the Assign phase when QA is enabled might be slightly slower than 3 parallel calls, but usually, LLM providers rate-limit parallel calls anyway, so the perceived speed difference will be negligible or even faster.
